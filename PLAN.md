# Outline
I want to build a series of agents that read through github issues for a repo and turn them into a PR. this is a multi-step, complicated process that will involve several agents with different tasks. I'm planning the pipeline as follows the think-act-observe process.

## 0. Setup (pre-flight, automated — no agent)
- Fetch issue title, body, and comments from GitHub API via `gh issue view`; write to ISSUE.md in run dir
- Clone repo to a temp directory if not already local (or verify local path is clean)
- Create a feature branch: `agent/<hash>` (e.g. `agent/a3f9c1b2`)
- Populate STATE.json including `branch_name` and `issue_body`
- This step is not an agent — it is deterministic Python run by the orchestrator

## 1. Plan (read-only)
- Contribution guidelines (README.md, CONTRIBUTING.md) will be set beforehand and passed to agent
- Read ISSUE.md (written by Setup) to ground the plan in the actual issue text
- Expand the given GH Issue by reading through repo and highlighting relevant files to change
- Create a PLAN.md that contains sequential list of changes to make
  - For each step, include a quick verification to ensure code works as intended
- Include an informal verification step for step 3. (e.g. run lint, build)
- Write no code in this step
- If looped from 2., 3., or 4.:
  - Read through failure report and revise the plan

## 2. Execute
- Translate text description from plan into code implementation
- Note justifications for updates in CHANGES.md

## 3. Validate
- Run verification steps passed from 1.
- Verify that code changes are in accordance to the spec generated by 1.
- If verification fails, classify failure, write report to VALIDATE.md (do not commit), and return result to orchestrator; routing is determined by the Looping section
- If verification passes, continue to 4.

## 4. Test
- Run changes against existing tests to ensure sufficient coverage
- Evaluate test results and append new tests as needed
- Test edits (not additions) are permitted only if the issue explicitly changes a public interface; justification required in TEST.md before editing
- Test deletions are never permitted — classify as `plan_invalid`, write report to TEST.md, and return result to orchestrator
- If tests fail and no edit is justified, classify failure, write report to TEST.md, and return result to orchestrator; routing is determined by the Looping section
- If tests pass: stage and commit all changes (source files from FILES.md + any appended test files) with message `agent: <issue title> (#<issue number>)`; write commit SHA to STATE.json

## 5. Summary
- Only runs when steps 1–4 have completed successfully
- Collect context from previous agents and format changes into PR
- Squash all commits on `agent/<hash>` into a single commit via `git rebase -i origin/main`
- If STATE.json contains no `pr_url`: create PR via `gh pr create`; write URL to STATE.json
- If STATE.json contains a `pr_url` (loop-back): force-push the squashed commit to the same branch — do not create a new PR
- Poll CI status after push; if CI fails, write failure report to SUMMARY.md and return result to orchestrator (global loop back to 1.)
- On success, write final SUMMARY.md with PR URL, CI status, loop count, and cost; continue to 6.

## 6. Report (post-flight, automated — no agent)
- Always runs — regardless of whether the pipeline succeeded or was aborted at any earlier step
- Closes the observability trace: writes TRACE.json with outcome, all spans, total tokens, and total cost
- Outcome is `pass` if `pr_url` is set and `ci_status == pass`; otherwise `fail`
- On `fail`: writes FAILURE.md to run dir containing `failure_source`, `last_failure_reason`, `current_step`, and `loop_count`
- Exports trace to Arize Phoenix (self-hosted) if `PHOENIX_COLLECTOR_ENDPOINT` is set
- This step is not an agent — it is deterministic Python run by the orchestrator


# Control

## Local Memory
0. Setup: ISSUE.md
1. Plan: PLAN.md, FILES.md
2. Execute: CHANGES.md
3. Validate: VALIDATE.md
4. Test: TEST.md
5. Summary: SUMMARY.md
6. Report: TRACE.json, FAILURE.md (on failure only)

## Shared State
- STATE.json
  - repo_url
  - local_dir: resolved path to `.agent/<hash>/` for this run
  - issue_url
  - issue_body
  - branch_name
  - loop_count: [0, 2]
  - local_loop_count: [0, 1]
  - current_step: [setup, plan, execute, validate, test, summary, report]
  - plan_version
  - last_failure_reason
  - failure_source: [exec, validate, test, ci]
  - ci_status: [pass, fail, None]
  - commit_sha: str | None
  - pr_url: str | None
  - cost_budget_usd: float (default 2.00)
  - cost_spent_usd: float (running total, updated after each agent call)
  - read_only: list of repo-relative file paths that agents may never write
    - Default (populated by Setup): README.md, CONTRIBUTING.md, LICENSE, .github/**, *.lock, .env*
    - Additional entries can be added by the Plan agent if it identifies files that must not change

## Looping

### Failure classification
- `minor`: lint error, compile error, trivial logic mistake → retry in local exec↔validate cycle (no re-plan)
- `spec_deviation`: code doesn't match PLAN.md spec → loop back to Execute only, inject VALIDATE.md
- `plan_invalid`: plan is contradicted by test results, or fundamentally wrong → loop back to Plan (global loop)
- `unrecoverable`: max loops reached, or agent explicitly signals it cannot proceed → route to Step 6 (Report writes FAILURE.md and exits non-zero)

### Loop caps
- Global loop (back to Plan): hard cap of 3
- Local exec↔validate cycle: cap of 2 per global iteration; resets to 0 on each global loop-back
- If `cost_spent_usd >= cost_budget_usd` before any agent call, set `failure_source = budget_exceeded` and route to Step 6
- Failure classification is the responsibility of the Validate and Test agents; they set `failure_source` in STATE.json

## Context Injection on Loop-back
- On any loop-back, the orchestrator builds a "Previous Attempt" block and prepends it to the agent's user prompt:
  ```
  ## Previous Attempt (loop N)
  Failure source: <failure_source>
  Failure reason: <last_failure_reason>

  ### Relevant artifacts
  <contents of VALIDATE.md or TEST.md or SUMMARY.md, whichever triggered the loop>
  ```
- The Plan agent always receives this block when `loop_count > 0`
- The Execute agent receives it when `failure_source == spec_deviation`
- Token budget for this block is capped at 4000 tokens; truncate oldest content first

## Tool Use

### execute_cli policy
- Allowlist is a set of (binary, allowed_args_prefix) pairs, not just binary names
  - e.g. `("pytest", "tests/")`, `("ruff", "check")`, `("python", "-m pytest")`
- Shell metacharacters (`;`, `|`, `&`, `$(`, `` ` ``) are rejected before execution
- Commands are run with CWD set to repo root; `..` path components are rejected
- Allowed binaries: `python`, `node`, `cargo`, `npm`, `pytest`, `ruff`, `mypy`, `git` (scoped args), `gh` (scoped args)

### Per-agent manifests
1. Plan
   1. ls
   2. read_file
   3. write_file (only PLAN.md and FILES.md)
   4. grep
2. Execute
   1. read_file
   2. write_file
   3. create_file
   4. execute_cli (allowlisted — see policy)
   5. grep
3. Validate
   1. read_file
   2. write_file (only VALIDATE.md)
   3. execute_cli (lint and build commands from PLAN.md verification steps only)
   4. grep
4. Test
   1. read_file
   2. append_file (test files only — no overwrite)
   3. execute_cli (test runner; git add + git commit on success)
   4. grep
5. Summary
   1. ls
   2. read_file
   3. grep
   4. execute_cli (git rebase, git push --force-with-lease, gh pr create/view)

## Agent File Storage
- All agent-produced artifacts live in `<repo_root>/.agent/<hash>/`
  - Artifacts: ISSUE.md, PLAN.md, FILES.md, CHANGES.md, VALIDATE.md, TEST.md, SUMMARY.md, TRACE.json, STATE.json, RUN.log, FAILURE.md (on failure only)
  - These are entirely separate from the project's own top-level files
- Hash is the first 8 hex characters of the SHA-256 of the issue URL
- Add `.agent/` to `.gitignore` if not present
- Agents never read or write outside this directory except for the target source files listed in FILES.md

# Observability
- Each agent call = one **span**; the full pipeline run = one **trace**
- Trace opens at Step 0 (Setup start); closes at Step 6 (Report)
- Each span includes: token input/output, cost, files read + written, tool calls, outcome
- TRACE.json written by Step 6 (Report); RUN.log written continuously throughout the run
- Outcome field in TRACE.json: `pass` = PR created + CI green; `fail` = FAILURE.md written (with reason)
- Export trace to Arize Phoenix (self-hosted) via OTLP if `PHOENIX_COLLECTOR_ENDPOINT` is provided (handled by Step 6)


# Evaluation

## Individual Agents
- LLM-as-judge rubric per agent:
  - Plan: (1) all referenced files exist, (2) each step is unambiguous and actionable, (3) verification commands are runnable
  - Execute: (1) all FILES.md entries were addressed, (2) CHANGES.md justification matches each edit, (3) no files outside FILES.md were modified
  - Validate: (1) all PLAN.md verification steps were run, (2) each pass/fail verdict has evidence, (3) failure classification is correct
  - Test: (1) no existing tests modified without justification, (2) new tests are syntactically valid, (3) all tests pass after appends
  - Summary: (1) PR description accurately reflects CHANGES.md, (2) PR is linked to the issue, (3) CI status is correctly reported
- Human feedback: binary pass/fail on the final PR (would you merge this?)

# Concerns
- **Execution safety**: Execute agent can write arbitrary files. Mitigated by scoping writes to FILES.md entries and the read_only list, but not fully sandboxed.
- **Cost runaway**: Multi-loop runs on large repos can be expensive. Mitigated by `cost_budget_usd` cap, but the cap is a soft guardrail.
- **GitHub rate limits**: Issue fetch and CI polling hit the API. Use `gh` CLI (which handles auth and retry) rather than raw HTTP.
- **Branch conflicts**: If `agent/<hash>` already exists from a prior failed run, Setup must decide whether to reset it or fail loudly.
- **Secrets exposure**: `.env*` and credential files must be in `read_only` by default. The allowlist on execute_cli reduces but doesn't eliminate risk.
- **Test suite absence**: Some repos have no tests. Test agent must handle this gracefully (write TEST.md noting absence, don't fail).
- **Large repos**: Plan agent may hit context limits scanning a large repo. FILES.md scoping is the primary mitigation; may need chunked grep strategies.


# Phase 6: Code Sandboxing (Docker)

In Step 0 (Setup), start a Docker container with the repo mounted at `/workspace`. Code execution commands (python, pytest, ruff, mypy, node, cargo, npm) route through `docker exec` inside the container. git and gh commands always run on the host. The container is torn down in Step 6 (Report).

## 0. Setup (updated)
- After branch creation and STATE.json initialisation, call `start_container(repo_root)` and `install_project_deps()`
- If Docker is not installed: print warning to stderr, continue without sandbox (soft-fail)
- Write `container_id` to STATE.json when a container is started

# Phase 7: Web Server

Expose the pipeline via HTTP. No changes to existing pipeline logic.

## Endpoints

- `GET /health` → `{"status": "ok", "version": "0.1.0"}`
- `POST /issue` → run pipeline, return result

### Request schema
```json
{ "issue_url": str, "repo_url": str, "local_path": str|null, "guidelines": str|null, "budget": float }
```

### Response schema
```json
{ "run_dir": str, "outcome": "pass"|"fail", "pr_url": str|null, "cost_spent_usd": float, "loop_count": int }
```

## Steps

**7.1 — `pyproject.toml`**
- Add `fastapi>=0.110.0` and `uvicorn[standard]>=0.29.0` to runtime dependencies

**7.2 — `server.py`** (new file, project root)
- FastAPI app with `GET /health` and `POST /issue`
- Inline `guidelines` string written to tempfile; cleaned up in `finally`
- `except SystemExit` converts `run_report`'s `sys.exit(1)` → `outcome="fail"`, HTTP 200
- Unhandled exceptions → HTTP 500

**7.3 — `main.py`** (restructured)
- Add argparse subparsers:
  - `run <issue_url> <repo_url> [--local-path] [--guidelines] [--budget]`
  - `serve [--host 127.0.0.1] [--port 8080]`
- Verification: `python main.py --help` shows both subcommands
